― Presence Logger のしくみ ―

カメラが「見た」は
どうやってDBに届くの？

カメラに人が映ってから、それが Oracle のデータベースに記録されるまで。途中に出てくる mosquitto や、なぜ二重登録されないのかを、絵で追いかけます。

所要 6分 / 中学生でも読めるように書いてあります
①

そもそも、何がしたいの？

USB カメラで「人がいる／いない」を見張って、入った（ENTER）と出た（EXIT）の瞬間だけをデータベースに記録したい。ただし条件がひとつ、とても厳しい。

それは「ちょうど1回だけ書く」こと。途中で電源が落ちても、WiFi が切れても、データベースが一時停止していても ── 復旧したあとに、同じ出来事を 2回書いてもダメ、0回（消える）でもダメ。きっかり1回だけ残す。専門用語で Exactly-once（イグザクトリー・ワンス）と言います。

📮 これは「大事な手紙を、確実に1通だけ届ける」のと同じ問題。届いたか不安で何度も同じ手紙を送ると、相手は同じ手紙を何通も受け取る。かといって送らなければ届かない。「届いたよ」という返事（受領確認）が来るまで送り続けつつ、受け取る側が「これは前にもらった手紙だ」と気づいて2通目は捨てる ── この仕組みがこのアプリの心臓部です。
②

登場するのは、この7人

このアプリは1個の大きなプログラムではなく、役割の違う小さな部品が協力して動いています。4つは Docker コンテナ（独立した小部屋）、2つはメモ帳（SQLite）、最後がゴール。

カメラを1枚ずつ見るdetector

毎秒数フレーム、映像を1枚ずつ AI に渡して「人が写っているか」とスコアを得る。この時点ではまだ何も記録しない。

camera.read() → PersonDetector.detect(frame)

3秒の「様子見」で本物だけ確定detector

一瞬の誤検知で記録すると ENTER/EXIT が乱れる。そこで状態機械(FSM)が「人がいる状態が 3秒続いたら 本当に ENTER」と判断。チラついた候補は取り消す（デバウンス）。

PresenceFSM.observe() → Transition(ENTER)

出来事に「背番号」を付けてメモ帳に書くdetector

確定した瞬間、世界で1つだけの ID（event_id＝UUID）を発行。これが手紙の追跡番号になる。先に自分のメモ帳(SQLite)に保存してから送る ＝ 送る前に必ず記録。

buffer.insert_pending(event_id=…) → 永続化

郵便局へ投函（QoS=2 で）detector

出来事を JSON にして mosquitto の presence/event という宛先（topic）へ publish。配達品質は最高ランクの QoS=2＝「重複なく、ちょうど1回配達」を指定する。

mqtt.publish_event(topic, payload, qos=2)

郵便局が bridge へ確実に配達mosquitto

mosquitto は宛先を購読(subscribe)している bridge へメッセージを渡す。bridge が一瞬落ちていても、QoS=2 の約束のもとで配達を完了させる。

presence/event ── 内部ネットワーク presence-net のみ

受け取ったら、まず自分のメモ帳へbridge

bridge は中身を検証して、すぐ inbox(SQLite)に status=received で保存。ここでも DB より先にメモ帳。だから直後に停電しても出来事は消えない。同じ event_id は2度書かない（重複防止の1つ目）。

inbox.insert_received(...) ON CONFLICT DO NOTHING

いまの WiFi から「書き込む先」を選ぶbridge

bridge は nmcli で接続中の WiFi の SSID を読み取り、profiles.yaml から対応する Oracle（工場A直接 / 工場B Wallet など）を解決。未知の SSID なら送らずに保留する。

network.get_current_ssid() → resolver.resolve(ssid)

Oracle に MERGE で書き込む（本番は Java通訳経由）bridgeoracle-jdbc

普通の INSERT ではなく MERGE を使うのがキモ。「同じ内容が既にあれば何もしない、無ければ入れる」。だから万一同じ出来事が2回届いても、2回目は0行追加＝無視される（重複防止の2つ目）。本番DBには Python から直接繋げないので、bridge は POST /merge で Java通訳(oracle-jdbc)に依頼し、通訳が JDBC で実際の MERGE を実行する。MERGE の中身も冪等性も変わらない。

client_mode=jdbc → POST /merge → JDBC MERGE → rows_affected = 1 or 0

「書けたよ」を返信し、完了マークbridge

MERGE 成功で inbox を sent に更新し、mosquitto 経由で detector に ACK（受領確認）を返す。detector は ACK を受けて初めて自分のメモ帳を acked にし、その出来事の旅が終わる。

mqtt.publish_ack(event_id, mk_date_committed) → buffer.mark_acked()

実際に流れる JSON は、こんな1行です（追跡番号 event_id が detector→bridge→ACK までずっと同じなのがポイント）：

💻 detector が郵便局へ投函するメッセージ
// presence/event に publish される中身（一部）
{
  "event_id": "e6ed87d4-1a92-4aa6-bbb2-129dc66c327b",  // 追跡番号
  "event": "ENTER",                 // 入った / 出た
  "event_time": "20260428113337",  // MK_DATE になる時刻
  "monotonic_ns": 1456585470085,   // 時計ズレ補正用の単調時間
  "wall_clock_synced": true,        // NTP同期できてるか
  "device_id": "raspberrypi5"
}
④

なぜ「ちょうど1回」になるの？

秘密は、たった3つの仕掛けの合わせ技です。1つでも欠けると二重登録や取りこぼしが起きます。

1

QoS=2 ── 郵便局の「重複なし配達」

MQTT の最高品質。ネットワークがどれだけ不安定でも、メッセージは 欠落せず・重複せず ちょうど1回 bridge に届く。配達途中の信頼性をここで担保。

2

SQLite に「送る前」「書く前」必ず記録

detector も bridge も、行動の前にメモ帳へ書く。だから途中で電源が落ちても出来事は生き残り、再起動後に「まだ返事が来ていないもの」を自動で再送できる（取りこぼし防止）。

3

Oracle MERGE ── 同じ内容なら「何もしない」

再送で同じ出来事が2回届いても、MERGE は既存行を見つけて 0行追加で素通り。テーブルにユニークキーを足さなくても重複が入らない（二重登録防止の最後の砦）。

3つ目の MERGE は、実際にはこんな SQL です。「キーが一致する行が無いときだけ INSERT」＝何度流しても結果が変わらない（これを冪等＝べきとうと言います）：

💻 冪等な書き込み（Python: oracle_client.py / Java通訳: Main.java 共通）
MERGE INTO HF1RCM01 t
USING (SELECT :1 AS MK_DATE, :2 AS STA_NO1, … FROM dual) s
ON (t.MK_DATE = s.MK_DATE AND t.STA_NO1 = s.STA_NO1
    AND t.STA_NO2 = s.STA_NO2 AND t.STA_NO3 = s.STA_NO3
    AND t.T1_STATUS = s.T1_STATUS)
WHEN NOT MATCHED THEN                -- 一致する行が無いときだけ
  INSERT (MK_DATE, STA_NO1, …, T1_STATUS, UPCMPFLG)
  VALUES (s.MK_DATE, s.STA_NO1, …, 1)   -- 2回目以降は 0 行 = 無視
🔭 点が線になる瞬間

3つの仕掛けは、それぞれ別の不安に答えています。

QoS=2「配達中に消えたり増えたりしない？」
SQLite「途中で停電したら忘れない？」
MERGE「再送で二重に入らない？」
↓ 3つを重ねると
どこで何が止まっても、復旧後に 「ちょうど1回」 が成立する ── これが Exactly-once の正体。
⑤

トラブルが起きても、止まらない

現場では Oracle が落ちたり、WiFi が圏外になったり、時計がズレたりします。bridge はそれぞれに「待つ」「保留する」「補正する」で対応し、出来事は inbox に溜めたまま失いません。

✓ 対応

DBが一時停止

書き込み失敗を再試行（間隔を 5秒→15秒→…と伸ばす指数バックオフ）。出来事は inbox に received のまま残る。

⚠ 守る

DBが恒久エラー（表が無い等）

サーキットブレーカが OPEN になり、無駄打ちを止めて15分後に再挑戦。CRITICAL ログで人に知らせる。

✓ 対応

知らないWiFiに繋がった

宛先が分からないので送らず保留(hold)。正しい SSID に戻ると、溜めた分をまとめて送る。

✓ 対応

時計がまだ合っていない

NTP同期前の出来事は「ブレない単調時間」で記録し、同期できた瞬間に正しい時刻へ後から補正して書く。

そして🔁 どのトラブルでも共通するのは「出来事は消えず inbox に残り続ける」こと。detector 側も ACK が来るまで再送し続けます。だから一時的な障害は、回復すれば自動で巻き戻されて続きから完了します ── 人が手で直す必要はありません。
⑥

もし郵便局が無かったら？

よくある疑問 ──「mosquitto を消したら全部壊れる？」。答えは半分だけ Yes です。正しさとラクさを分けて考えると、何が残って何が消えるかがはっきりします。

まず大事な結論から。「ちょうど1回」という“正しさ”は、郵便局が無くても保たれます。それを守っているのは郵便局ではなく、両端と DB に仕込んだ仕掛け ── 背番号(event_id)＋控え(SQLite)＋ACK再送＋MERGE ── だからです。

✉️ 手紙に「No.7」と書いて控えをノートに残す。「No.7 届いた」の返事が来るまで同じ No.7 を何度でも送る。受け手は「No.7 はもう受け取った」と気づけば2通目を捨てる。── 途中で消えても再送、二重に届いても捨てる。だから運び役が郵便局でも雑な土管でも、記録に残る No.7 はちょうど1回。
等式🧮 最低1回は届ける（at-least-once）＋ 重複は捨てる（冪等）＝ 実質ちょうど1回（exactly-once）。運び役には「いつかは届く」だけを求め、「重複させるな」は受け手の背番号チェックと MERGE が吸収する。だから MQTT の QoS=2 は“配達区間の上乗せの安心”であって、正しさの土台ではありません。

では、郵便局を消すと何を失う？ ── 失うのは「正しさ」ではなく “ラクさ”。「ポストに入れるだけ」だった毎日が、ぜんぶ手作業に変わります。

🏠① 相手の居場所
🚫 郵便局ナシ相手の住所も、いま起動しているかも自分で把握。相手が再起動中だと接続拒否。
↔
📮 郵便局アリポスト（topic）に入れるだけ。相手が誰でも・どこでも・寝ていてもOK。
📨② 返事（ACK）の受け取り
🚫 郵便局ナシ返事を受けるため、自分の家にも受付窓口を常駐。出す側なのに受ける設備まで持つ。
↔
📮 郵便局アリ返事も同じ郵便局が運ぶ。出すだけ・購読するだけでいい。
🚚③ 配達そのものの信頼性
🚫 郵便局ナシ再送の間隔・重複の弾き方・順序を自分で実装（バグの温床）。
↔
📮 郵便局アリQoS=2 が肩代わり。書留と同じ安心を最初から。
🔌④ 受け手を増やす
🚫 郵便局ナシ出し手×受け手の数だけ専用ホットライン。1人増えると全員が配線変更。
↔
📮 郵便局アリtopic を購読するだけ。監視ツールが横から覗くのも自由。
📦⑤ 別マシンへ引っ越し
🚫 郵便局ナシ配置が変わるたびつなぎ方を全部書き直し。
↔
📮 郵便局アリポストに入れる動作は同じ。K3s で分散してもラク。
🧭 「正しさ」と「ラクさ」は別の係

この2つを混ぜて考えると「郵便局を消したら全部壊れる」と誤解しがち。実は担当が違います。

正しさ係「背番号＋控え(SQLite)＋ACK再送＋MERGE」
ラクさ係「mosquitto」
↓ だから
郵便局を消しても “ちょうど1回”は守られる。失うのは“ラクさ”だけ ── でも本気で作り直すと、結局もう一度 mosquitto ができあがる。

ひとことで言うと

カメラの「見た」は、郵便局(mosquitto)を経由し、両側のメモ帳(SQLite)に守られながら、冪等な書き込み(MERGE)で Oracle にちょうど1回だけ届く。

  • detector が見て・3秒様子見して ENTER/EXIT を確定し、追跡番号付きで投函する
  • mosquitto が QoS=2 で「重複なく確実に」bridge へ配達する
  • bridge が inbox に記録 → WiFi で宛先を選び → MERGE で Oracle に書く → ACK を返す
  • 「ちょうど1回」は QoS=2 ＋ SQLite永続化 ＋ MERGE冪等 の3点セットで成立する
  • 障害が起きても出来事は inbox に残り、回復後に自動で続きから完了する
  • 正しさ（ちょうど1回）は SQLite＋ACK＋MERGE が守るので郵便局に依存しない。mosquitto が守るのはラクさ（疎結合・双方向・拡張・分散）
