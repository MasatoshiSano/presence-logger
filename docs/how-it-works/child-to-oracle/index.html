どうやって Oracle にあがる？

「どんな形で・どこに送れば・どう記録されるの？」を、入口から台帳まで一本の道として追いかけます。

所要 5分 / 中学生でも読めるように書いてあります
①

まず、ぜんたいの道すじ

子ラズパイは Oracle に直接は触りません。無線で「ハブ役のラズパイ」に渡し、ハブが代表してOracleに書き込みます。

送るものは「CSVの1行」を JSON に直したもの。送り先は ハブの MQTT（10.42.0.1:1883）。あとはハブの中の係（bridge）が、工場ネットワーク経由で Oracle の台帳テーブル HHC001 に転記します。

🏭 工場の例え。各持ち場（子ラズパイ）が伝票（CSV）を書く → 集荷係（MQTT）が本社の郵便受けに集める → 本社の事務（bridge）がまとめて台帳（Oracle）に転記する。持ち場は台帳を直接いじらない＝鍵（工場ネット）を持つのは本社だけ、という分担です。

③

どんな形で送る？

子Pi内部では CSVの1行。MQTTを流れるときは JSON。最後はOracleの1行。形は変わっても中身は同じ5項目です。

📄 子Pi内のCSV（1行＝1件・複数行OK）
# YYYYMMDDhhmmss , STA_NO1 , STA_NO2 , STA_NO3 , T1_STATUS
20260610173000,100,200,300,3
20260610173100,101,201,301,1
📨 MQTTで送るJSON（発送係が変換）
{
  "event_id": "<行から自動生成のID>",   // 再送しても二重にならない鍵
  "mk_date": "20260610173000",     // → Oracle MK_DATE
  "sta_no1": "100", "sta_no2": "200", "sta_no3": "300",
  "t1_status": 3,                  // → Oracle T1_STATUS
  "device_id": "child-01",
  "schema_version": 1
}

CSVの位置と、Oracleの列名の対応：

1番目「20260610173000」
→
日時 MK_DATE
2〜4番目「100,200,300」
→
席番号 STA_NO1 / STA_NO2 / STA_NO3
5番目「3」
→
状態 T1_STATUS
（固定値）
→
更新フラグ UPCMPFLG = 1
ヒント💡 在席の ENTER/EXIT とは別の経路です。在席は席番号をハブ側で決めますが、こちらは席番号も状態もCSVの値をそのままOracleに書きます。
④

Oracleにあがるまでの流れ

投函してから台帳に載るまで、6ステップ。

子PiをAPに接続

子ラズパイをハブの無線「presence-hub」に繋ぐ。下の接続情報を使う。自動で 10.42.0.x のIPがもらえる。

sudo nmcli device wifi connect "presence-hub" password "presence-hub-2026"

CSVを置く

送りたい行を書いたCSVを子Piの outbox フォルダに置く。

~/outbox/records.csv

発送係がJSONで投函

child-csv-to-mqtt.py が各行をJSONにして MQTT の presence/record へ送信（QoS2）。送ったファイルは sent へ退避。

MQTT_HOST=10.42.0.1 python3 child-csv-to-mqtt.py

ハブが受け取り控える

ハブのmosquittoが受信→bridgeが「record_inbox」に保存。ここまでは工場網が無くてもOK（貯めておける）。

log: record_received

Oracleへ転記

ハブが工場網（HIME-H-REAP）に繋がっている間に、bridgeが行の値そのままで HHC001 に書き込む（MERGE）。

log: record_committed / rows_affected:1

完了の合図

書けたら presence/record/ack で「受領」を返す。失敗時は時間をおいて自動で再挑戦。

🔑 presence-hub の接続情報（子Pi側で使う）
# Wi-Fi（ハブのドングルAP）
SSID     : presence-hub        # 2.4GHz・隠しではないので一覧に出る
パスワード : presence-hub-2026   # WPA2。運用で変えたら↓も更新
ハブ(GW) : 10.42.0.1           # 子Piは 10.42.0.x が自動付与(DHCP)

# ① 接続（子ラズパイで1回）
sudo nmcli device wifi connect "presence-hub" password "presence-hub-2026"

# ② つながったか確認
ping 10.42.0.1            # ハブに届けばOK
nmcli device wifi list | grep presence-hub   # 電波が見えるか
前提⚠️ この電波が出ているのは、ハブ側でAP（presence-hub）が起動しているときだけです。ハブで Desktop「HIME-H-REAP 接続」を押すと、APの起動とMQTTの公開まで自動で整います。パスワードはハブの secrets.env の WIFI_AP_PSK で管理しているので、変更したらこの資料も直してください。

確認のしかたと注意点

本当にOracleへ入ったかは、Desktopの「直近30件の記録」で直接DBを見て確かめられます。

✓ できる

DBを直接確認

「直近30件の記録」アイコンが HHC001 を直接SELECT。実際に書けた行が最新順で見える。

✓ できる

二重防止（冪等）

同じ行を再送しても event_id とDBの鍵で1件にまとまる。安心して再送できる。

✓ できる

通信断に強い

工場網が切れても record_inbox に貯まり、復旧後にまとめて書き込む。

⚠ 条件

Oracle書込みは工場網が要る

Oracleは工場網(HIME-H-REAP)経由でしか届かない。未接続の間は「貯め」だけで送信は保留。

✓

「直近30件」は席番号で絞る

このプロファイルの席（996/995/994）で絞って表示。別席のレコードは出ない。

!

本番テーブルに直接入る

HHC001 は本番。動作確認で投げたダミーは消さないと残る（後始末用に範囲削除あり）。

注意⚠️ 子Piが書き込めるのは、ハブで mosquitto を子Pi向けに公開しているときだけ（10.42.0.1 のみにバインド＝工場網には晒さない）。Desktopの「HIME-H-REAP 接続」を押すと、AP起動とこの公開まで自動で整います。
🔭 なぜ「JSONでMQTT」なの？

生のCSVをそのまま流すより、JSONにすると ID（再送しても二重にならない）・送り主・形式バージョン を一緒に運べます。CSVは子Piの中の入力ファイル、ワイヤ（通信）上はJSON、と分けるのが堅い設計です。

だから子Piは「CSVを置くだけ」、信頼性の面倒は発送係とbridgeが見る。

ひとことで言うと

子Piは「CSVの行」をJSONにしてハブのMQTT（10.42.0.1 / presence-record）へ投函すれば、あとはハブのbridgeがOracle(HHC001)へ転記してくれる。

  • 形：CSV 1行（日時, 席1, 席2, 席3, 状態）→ JSON → Oracleの1行
  • 送り先：AP「presence-hub」に繋いで MQTT 10.42.0.1:1883 の presence/record
  • あがり方：mosquitto → bridge（record_inbox に控え）→ 工場網経由で Oracle へ MERGE
  • 確認：Desktop「直近30件の記録」で HHC001 を直接見る
  • 強み：再送しても二重にならない／通信断でも貯めて後で送る
― おわり ―
presence-logger / 子ラズパイ → MQTT → bridge → Oracle の解説（社内向け）
