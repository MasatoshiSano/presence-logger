― presence-logger を 2 台に ―

2台目のラズパイは
どうやってつなぐ？

カメラをもう1台増やしたい。でも「データベースに二重で書いたり、片方のイベントが消えたりしない？」が心配。その不安を順番にほどきます。

所要 5分 / 中学生でも読めるように書いてあります
①

増やすのは「目」だけ。「事務所」は1つのまま

presence-logger は中で 4 つの部品が動いています。2台目で増やすのは、そのうち カメラを見張る部分（detector）だけです。

1台目はこれまで通り全部入りで動かします。データベース（Oracle）に書き込む係＝bridge は、ぜったいに1つだけ。2台目には目（detector）だけを置いて、1台目の事務所へ報告を送らせます。

🏢 会社で考えると、事務所（bridge＋DB）は本店に1つ。新しく見回りスタッフ（2台目のカメラ）を雇っても、報告書を出す窓口は本店のまま。スタッフが2人になっても、台帳に記入する人を2人にしてはいけません。記入係が2人いると同じ台帳を取り合って混乱するからです。
②

登場人物（だれが何を持つか）

2台構成になると、こういう役割分担になります。色つきカードで確認しましょう。

🏠
1台目 ＝ 本店

フル装備

mosquitto（窓口）・detector・oracle-jdbc・bridge の4つ全部。DBに書くのはここの bridge だけ。

mosquitto + detector + oracle-jdbc + bridge
📷
2台目 ＝ 見回り

detector だけ

カメラを見張って ENTER/EXIT を見つけ、1台目の窓口へ送るだけ。DB には触らない。

detector のみ（MQTT_HOST=1台目のIP）
📮
共有の窓口

mosquitto

1台目で動く1つの broker。2台のカメラが同じ presence/event に投函し、ACKを受け取る。

presence/event ・ presence/event/ack
📮 窓口は1つ（1台目の mosquitto）
tcp://<1台目のIP>:1883
📷 1台目 detector
station = 996/995/994
→ 同じ窓口へ投函
📷 2台目 detector
station = 別の番号
→ 同じ窓口へ投函
③

壊さないための「3つの約束」

この設計はもともと複数カメラを想定しています。安全に増やせるかは、次の3点だけで決まります。

A

station 番号を必ず別にする

DBの行の見分けは MK_DATE+STA_NO1/2/3 の組。2台が同じ番号だと同じ行を取り合って上書きします。2台目は device.yaml で別の三つ組にする。これが一番大事。

✓

event_id の重複 ── 何もしなくてOK

イベントIDは uuid.uuid4() で作られます。世界中で被らない乱数なので、2台でも自動的に一意。ACKを取り違える心配はありません。

!

bridge は増やさない

2台目で bridge を動かすと、DBへの書き込み係が2人になり二重書き込みの原因に。2台目は detector だけ起動する。

ヒント💡 ACK（受領通知）は共有の presence/event/ack に流れ、各 detector は自分のIDのACQだけ拾う仕組みです。IDが乱数で被らないので、2台目のACKを1台目が誤って消す、といった事故は起きません。
④

実際の手順

「1台目を少し開ける → 2台目に detector だけ置く」の2部構成です。

1台目：窓口をLANに開ける

いまの mosquitto はホストにポート非公開（コンテナ内だけ）。2台目から届くよう docker-compose.yml の mosquitto に ports を足して再起動します。

ports: ["1883:1883"] を追加 → sudo docker compose up -d mosquitto

1台目：1台目のIPを確認

2台目から指す宛先。固定IP推奨。

hostname -I # 例: 192.168.10.20

2台目：リポジトリを置いて install

1台目と同じ手順でセットアップ。設定ファイルが /etc/presence-logger/ に作られます。

sudo scripts/install.sh

2台目：station 番号を別にする

device.yaml の station 三つ組を1台目と違う値に。これが約束Aです。

sudo $EDITOR /etc/presence-logger/device.yaml

2台目：detector だけ起動（宛先は1台目）

2台目専用の compose ファイルを使う。これには detector しか定義していないので、bridge等が連れて起動する心配がありません。MQTT_HOST に1台目のIPを渡します。

MQTT_HOST=192.168.10.20 sudo -E docker compose -f docker-compose.detector.yml up -d

確認：本店の台帳に届いたか

1台目の bridge ログに2台目のイベントが出て、DBに別stationの行が増えればOK。

sudo cat /var/log/presence-logger/bridge.log | grep merge_committed
💻 2台目だけ変える2か所
# /etc/presence-logger/device.yaml ── station を別番号に（約束A）
station:
  sta_no1: "886"   # 1台目(996)と別にする
  sta_no2: "885"
  sta_no3: "884"

# 起動 ── 2台目専用 compose（detectorのみ定義）＋ 1台目IPを MQTT_HOST に
MQTT_HOST=192.168.10.20 sudo -E docker compose -f docker-compose.detector.yml up -d
# topic は触らない: presence/event / presence/event/ack のまま共有
✓ 2台目で動かす

detector

カメラ監視＋イベント送信。SQLite outbox も2台目ローカルに持つ。

⚠ 2台目では動かさない

bridge / oracle-jdbc / mosquitto

DB書き込みと窓口は1台目に集約。2台目で起動しないこと。

⑤

ポートを開ける前に ── 安全のために

窓口をLANに開けるということは、その線につながれる人なら誰でも投函できる、ということ。1点だけ必ず手当てします。

注意⚠️ mosquitto は今 allow_anonymous true（認証なし）。コンテナ内だけだから許されていた設定です。LANに公開するなら、パスワード認証を足すか、信頼できる閉じたネットワーク/VLANに限定してから開けてください。無認証のまま広いネットに晒すと、偽のENTER/EXITをDBに入れられます。
password_file を足す または閉域/VLAN限定 できればTLS 広いネットに無認証で晒さない

ひとことで言うと

2台目には detector だけを置き、station 番号を別にして、1台目の窓口（broker）へ報告させる。それだけで安全に増やせます。

  • 増やすのは目（detector）だけ。bridgeとDBは1台目に1つのまま。
  • station 三つ組を必ず別にする（同じだとDBの行を取り合う）。
  • event_id は uuid4 で自動的に一意。ここは何もしなくてよい。
  • 1台目で broker のポートを開ける前に認証 or 閉域を用意する。
